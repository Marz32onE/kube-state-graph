1. 專案定位

kube-state-graph 是一個 Go HTTP API，只交付 graph API server。 它從單一集中式 VictoriaMetrics（Prometheus 相容 HTTP API）讀取：

  • 拓樸指標：kube_*（由 kube-state-metrics 產出，帶 cluster external label）
  • 服務圖指標：traces_service_graph_request_total（由 Tempo / Alloy servicegraph 等產出）

在記憶體中 join 成多叢集的 pod / node / PVC / service / external / others 圖， 並輸出 Cytoscape.js 或 Grafana Node Graph JSON。 不使用 Kubernetes client-go、不 scrape KSM、不 ingest trace。

交付物邊界 ───────────── 本 repo 交付： kube-state-graph 二進位 + OpenAPI + 文件 + 測試 harness 本 repo 不交付： VictoriaMetrics、kube-state-metrics、service-graph producer、生產 K8s 部署（除範例 Helm/Docker） 典型部署拓樸 ──────────── cluster A ── KSM + service-graph ──┐ cluster B ── KSM + service-graph ──┼──► VictoriaMetrics ◄── kube-state-graph (:8080) cluster N ── KSM + service-graph ──┘ ▲ └── 僅 PromQL HTTP，單一 --prom-url

2. 資料流與依賴

2.1 上游如何餵資料

來源指標族關鍵 label用途
kube-state-metrics kube_pod_info、kube_node_info、PVC、Service、EndpointSlice 等 cluster（vmagent/Prometheus external_labels） 節點、pod-runs-on-node、pod-mounts-pvc、D29 連線字串解析索引
Service graph producer traces_service_graph_request_total cluster（trace 來源／client 端）、client_k8s_pod_uid、server_k8s_pod_uid pod-calls-pod；server 端 cluster 由全域 PodsByUID join 還原
Prometheus 原生 up — 區分「視窗內無拓樸」是 retention 問題還是上游故障

2.2 每次建圖的 PromQL fan-out

internal/build/topology.go 以 errgroup.WithContext 並行執行最多 8 條拓樸查詢；ReadServiceGraph 再跑 1 條 service-graph 查詢。查詢模板定義於 internal/promql/queries.go。

Query 常數包裝方式（概念）
kube_pod_info 等 KSM 系列last_over_time(<metric>[<window>]) @ <end>；可選 --metric-prefix
traces_service_graph_request_totalrate(...[<window>]) @ <end> + D30 sentinel 排除 user|unknown
/v1/clusters固定 1h lookback 的 cluster discovery（非請求視窗）

2.3 本機驗證 harness（非 CI 必跑）

local/kind/：Kind + in-cluster VM + KSM + Beyla/Alloy service graph + Grafana。用 make kind-up / make smoke 手動驗證。跨叢集情境由 internal/integration/（testcontainers VictoriaMetrics + 合成 series）覆蓋。

3. 請求生命週期（以 GET /v1/graph 為例）

HTTP GET /v1/graph?start=&end=&cluster=&... │ ▼ Gin middleware：request-id、logging │ ▼ /v1/* ：otelgin（僅 v1 路由有 trace）、API key、span enrich │ ▼ parseGraphRequest ── 驗證 start/end（RFC3339 或 Unix 秒）；僅強制 end > start │ ▼ context.WithTimeout(ctx, --build-timeout) ← internal/api/handlers.go runBuild │ ▼ Builder.Build(window, end) ├─ ReadTopology（errgroup，5～8 條 PromQL） ├─ ReadServiceGraph（1 條 PromQL + 與 topology join） └─ assemble → graph.NewGraph（含 adjacency） │ ▼ graph.Project(g, scope) ← 過濾與 BFS 在此，不在 build 階段 │ ▼ serialiseCytoscape / serialiseGrafanaNodeGraph │ ▼ JSON { apiVersion, clusters, elements }（確定性排序）

4.1 圖模型（核心不變式）

  • 節點 ID：pod <cluster>/<uid>；node <cluster>/<node>；PVC <cluster>/<ns>/<claim>；service <cluster>/<ns>/<svc>；others others/<label>；external external/<label>
  • 邊 ID：UUIDv5，輸入 <type>|<source>|<target>（固定 namespace，golden test 依賴）
  • labels：嚴格 map[string]string；數值指標（rate、p99）延後到未來 typed 欄位
  • ipaddress：頂層 data.ipaddress，不在 labels 放 pod_ip 等
  • GraphNode：密封介面（isGraphNode() 未匯出），序列化透過方法而非 type switch

5. 設計決策與好處

以下對應 design.md 的 D1–D31 精簡說明。

D1 單一 VictoriaMetrics upstream

為什麼：符合集中式可觀測性部署；一次 PromQL 可跨叢集；跨叢集邊的兩端在同一結果集內可 join。

好處：連線與查詢邏輯簡單；避免 N 個 per-cluster URL 與 informer 的歷史查詢缺口。

D2/D6 每次請求即時建圖、無 in-process cache

為什麼：v1 刻意保持小；時間旅行查詢與 cache 鍵對齊曾綁在一起，移除 cache 後也移除視窗對齊。

好處：語意直觀（所見即該視窗）；水平擴展時可把 cache 做成 Redis / materialiser 而不推翻投影模型。

代價：上游 QPS 與 HTTP 流量線性相關；需靠 VM 搜尋限制與 HPA 扛載。

D3 叢集範圍 ID

Node 名、PVC 名跨叢集會碰撞；pod 雖 UID 全域唯一，仍加 cluster/ 前綴使 log/JSON 自描述。

D7 過濾在投影層、不推進 PromQL

每次 build 載入 VM 中所有叢集；graph.Project 套用 cluster、namespace、edge_type、name、BFS traversal。好處：同一 *Graph 可服務多種 filter 組合（未來 cache 可共用 materialised graph）。

D27/D29/D30 端點解析與 sentinel

  • 空 UID + 含 :// → 連線字串解析（service / headless pod / others）
  • 空 UID + 非 URL → external 節點
  • PromQL 層排除 client/server = user|unknown（精確比對）

D31 Cytoscape compound nodes

僅 serialiseCytoscape 合成 type=cluster 群組與 parent 巢狀；核心 *Graph 不變。Grafana 路由保留 pod-runs-on-node 邊（無法巢狀）。

依賴刻意精簡

執行期主要：Gin、prometheus/client_golang、google/uuid、x/sync、OpenTelemetry。無 client-go——降低權限面與運維複雜度，代價是完全依賴指標合約正確。

6. 是否符合 Go 慣例

6.1 做得好的部分

  • layout 標準 cmd/ + internal/，模組路徑 github.com/marz32one/kube-state-graph
  • interfaces 小介面 promql.Querier、auth.Validator、clock.Clock + mockery 產生 mock，利於單元／元件測試
  • errors 具型別 build.Reason 映射 HTTP，而非字串比對
  • context 建圖與上游查詢貫穿 context.Context；逾時用 WithTimeout
  • concurrency errgroup.WithContext 並行 PromQL，取消會傳播
  • logging 標準庫 log/slog，結構化欄位
  • tools Go 1.24+ tool 指令釘選 swag、mockery；CI 用 go tool
  • testing table-driven、-race、property test（internal/graph/property_test.go）、golden JSON
  • docs 套件 doc.go 說明邊界；handler 上 swag 註解

6.2 刻意偏離或待對齊之處

  • 無 /v1 以外的版本目錄：v1 路由承諾長期穩定；破壞性變更應走 v2（D14）
  • Gin 而非 stdlib net/http mux：團隊選型，常見於需要 middleware 生態的 API
  • 密封介面：Go 慣用未匯出 marker method，非 Java sealed class——符合慣例
  • 生產碼無 test-only setter：依賴注入走介面，符合專案規範

7. 如何使用

7.1 建置與執行

make init          # 首次：模組 + golangci-lint + govulncheck
make build
./bin/kube-state-graph \
  --prom-url=http://localhost:8428 \
  --listen-addr=:8080 \
  --build-timeout=15s

7.2 重要設定

Flag / 環境變數預設說明
--prom-url / KSG_PROM_URLhttp://localhost:8428VictoriaMetrics Prometheus API
--listen-addr:8080HTTP 監聽
--build-timeout15s/v1/graph* 建圖逾時
--api-timeout5s/v1/clusters、/readyz
--api-keys-file空啟用 X-API-Key；可週期熱重載
--metric-prefix空僅 KSM 形狀指標加前綴；不影響 service-graph / up
OTEL_EXPORTER_OTLP_*未設未設則 tracing 為 no-op

7.3 HTTP API 速查

路由需 API key？用途
GET /v1/graph有設定時要Cytoscape JSON；支援 filter + traversal
GET /v1/graph/nodegraph同上Grafana Node Graph 形狀
GET /v1/clusters同上即時叢集列表
GET /v1/edge-types同上靜態邊類型目錄（與 builder 同步）
GET /livez、/readyz、/metrics否探針與自監控
/openapi.*、/docs否Scalar API 文件

7.4 查詢範例

end=$(date -u +%s)
start=$((end - 300))

# 叢集列表
curl -s "http://localhost:8080/v1/clusters" | jq .

# 全圖（最近 5 分鐘）
curl -s "http://localhost:8080/v1/graph?start=${start}&end=${end}" | jq '.clusters, (.elements.nodes | length)'

# 只看某叢集 + pod 呼叫邊 + 從某 pod 往外 2 hop
curl -s -G "http://localhost:8080/v1/graph" \
  --data-urlencode "start=${start}" \
  --data-urlencode "end=${end}" \
  --data-urlencode "cluster=prod-eu" \
  --data-urlencode "edge_type=pod-calls-pod" \
  --data-urlencode "root=prod-eu/8f8d4f1a-1234-4abc-9def-0123456789ab" \
  --data-urlencode "depth=2" \
  --data-urlencode "direction=out"

# 有 API key 時
curl -s -H "X-API-Key: ${KSG_API_KEY}" "http://localhost:8080/v1/graph?start=${start}&end=${end}"

7.5 上游部署檢查清單

  1. 每叢集 remote-write 前寫入 cluster=<name> external label
  2. KSM 匯出必要 kube_pod_info、kube_node_info
  3. Service graph 帶 client_k8s_pod_uid / server_k8s_pod_uid（建議 Alloy dimensions 含 k8s.pod.uid）
  4. EndpointSlice 若要做 D29：KSM --metric-labels-allowlist=endpointslices=[kubernetes.io/service-name]
  5. 詳見 docs/operations.md、docs/others-substitution.md

8. 如何加／改功能

8.1 建議流程（OpenSpec）

  1. 非平凡行為變更：在 openspec/changes/<name>/ 寫 proposal → design → specs → tasks
  2. openspec validate <name>、實作後 openspec verify
  3. 完成後 openspec archive 併入 openspec/specs/

8.2 新增邊類型（範例 checklist）

  1. 在 internal/graph/registry.go 的 EdgeTypes 註冊
  2. 在 internal/build/ 從新 PromQL 或既有 topology 組邊
  3. 更新 /v1/edge-types（自動來自 registry，勿與 builder 脫節）
  4. 元件測試 internal/api/*_test.go；必要時 golden：go test ./internal/api/ -update -run Golden
  5. 若改 PromQL 常數名：確認 self-metrics 的 query= label 穩定性
  6. make docs 更新 OpenAPI

8.3 新增 HTTP 查詢參數或過濾

  1. 擴充 graph.Scope（internal/graph/scope.go）
  2. 在 graph.Project 實作過濾邏輯（注意 edge re-add 規則，見 D7）
  3. parseGraphRequest 解析參數；swag 註解與 docs/api.md

8.4 修改 PromQL／指標合約

  1. 改 internal/promql/queries.go 與 Renderer
  2. 同步 design.md、integration fixture、docs/operations.md exporter 表
  3. service-graph 新 metric 必須重用 serviceGraphSentinelSelector 片段（D30）

8.5 修改介面時

編輯 .mockery.yaml 列出的介面後執行 make mocks 並提交 internal/*/mocks/；CI 有 mocks-drift job。

8.6 常見開發指令

make test                    # 全套件，-race -shuffle
make lint && make vet && make vuln
go test ./internal/graph/ -run TestProject -v
go test ./internal/api/ -run TestGolden -v
make kind-up && make smoke   # 本機 Kind（需 Docker）

11. 改進空間

11.1 架構／產品

  • 分散式 cache（已規劃）：Redis L2、background materialiser 或 graph DB；需重訪 TTL、cache key、D5 對齊策略
  • 延後的邊屬性：rate、p99_ms、error_rate、cross_cluster boolean——目前刻意不放進 labels
  • Per-tenant / RBAC：v1 僅共享 API key，無 per-cluster 授權
  • VM multi-tenant：v1.1 逃生閥，非現行模型

11.2 程式與營運

  • 無 in-process 併發上限：高流量時完全依賴 HPA；極端情況可能同時壓垮 VM（可觀察 kube_state_graph_* metrics 再決定是否恢復 semaphore）
  • 大型圖記憶體：百萬節點規模需 materialiser + 分片，非現行 in-memory *Graph 目標
  • 重複 UID 異常：PodsByUID 採 first-wins，可考慮 metric 或 log 告警

11.3 文件與範例漂移（建議修正）

README / README.zh-tw 與現行程式可能不一致：
  • 仍提及 ETag / If-None-Match / 304——v1 已移除 HTTP cache validator（見 tasks.md §32、design.md D6）；internal/api 已無 ETag 實作
  • 仍提及 --max-window / --max-skew——已改由 VM 搜尋限制負責（tasks.md §27 進行中／部分完成）
  • examples/python/get_graph.py 仍示範 ETag revalidation，與現行 API 不符
  • docs/api.md 若仍寫 ETag，應與 spec 同步刪除

11.4 測試與 DX

  • 完成 tasks.md §27 剩餘項（config 精簡、文件掃尾）
  • §32.D.4 手動驗證 Kind + Scalar：ipaddress 欄位、graph 路由無 ETag
  • 可加：make serve-docs 一鍵開啟本 HTML + Scalar

11.5 Go 風格微調（非必須）

  • 長期可評估 stdlib net/http + httprouter 減少 Gin 依賴（成本高、收益有限）
  • 對外穩定 API 可考慮產生 client SDK（OpenAPI → codegen）
