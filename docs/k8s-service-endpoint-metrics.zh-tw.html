Kubernetes Service / Endpoint 指標與圖譜建構

說明 kube-state-graph 為建構服務拓撲圖所需的三個 kube-state-metrics（KSM）系列、 各系列使用的 label、以及這些資料在圖譜中的用途。內容依據設計文件 D29 與原始碼實作整理。

目錄
為什麼需要 三個指標系列 Label 對照表 Topology 索引 連線字串解析 圖譜輸出 PromQL 查詢 KSM 設定 缺失時行為 相關指標

為什麼需要 Service / Endpoint 指標？

kube-state-graph 的主要呼叫關係來自 traces_service_graph_request_total（Grafana Alloy / Tempo 的 servicegraph connector）。 當某個 endpoint 的 pod UID 為空，但 client 或 server label 是 連線字串（含 ://，例如 mongodb://payments-svc.shop.svc.cluster.local:27017）時， API 必須把 DNS host 解析成叢集內的 Kubernetes 資源，而不是一律當成外部節點。

這就是設計決策 D29（connection-string resolution） 要解決的問題：

  • 兩段式 DNS（<service>.<namespace>）→ 解析為 type="service" 節點
  • 三段式 DNS（<pod>.<service>.<namespace>，Headless Service）→ 解析為真實後端 Pod
  • 無法解析 → 退回 type="others" 節點（保留原樣 label 字串）

Pod、Node 等基礎拓撲來自 kube_pod_info 等既有指標； Service 與 EndpointSlice 指標則專門用來建立「Service → 後端 Pod」的解析索引， 並在需要時按需物化 service-selects-pod 邊。

重要：這三個指標不會把叢集內所有 Service 都畫進圖裡。 它們只建記憶體索引；只有被 service-graph 連線字串實際引用到的 Service 才會變成圖節點， 以避免圖過於龐大。

三個指標系列

資料來源為 kube-state-metrics，由 topology reader 在每次 /v1/graph 請求時平行查詢。

1. kube_service_info

用途：記錄每個 Service 的基本屬性，主要是 cluster_ip。

  • 建立 ServicesByNameNS 索引：以 (cluster, namespace, service) 為 key
  • 連線字串解析到 Service 層級時，物化 type="service" 節點的 ipaddress
  • cluster_ip="None" 表示 Headless Service，此時不輸出 ipaddress

2. kube_endpointslice_labels

用途：把 EndpointSlice 對應到其所屬的 Service 名稱。

  • 讀取 label_kubernetes_io_service_name（KSM 將 Kubernetes label kubernetes.io/service-name 扁平化後的名稱）
  • 以 (cluster, namespace, endpointslice) join，得到 slice → service 對照表
  • 預設 KSM 不匯出此 label，需額外 allowlist（見下方設定章節）

3. kube_endpointslice_endpoints

用途：列出每個 EndpointSlice 中的各個 endpoint，並解析其後端 Pod。

  • 透過 targetref_kind / targetref_name / targetref_namespace 對應到 Pod
  • 只處理 targetref_kind="Pod"（或空值視為 Pod）的 endpoint
  • 保留 hostname，供 Headless Service 三段式 DNS 比對（StatefulSet 自訂 hostname 等情境）
  • 結合 slice→service 對照，建立 EndpointsByService 索引

Label 對照表

以下 label 為 exporter 相容性合約的一部分；名稱必須一致，否則 join 會失敗。

kube_service_info

Label是否必要圖譜建構用途
cluster 是 多叢集隔離；索引 key 的第一段；連線字串解析限定在 trace 來源叢集
namespace 是 索引 key；Service 節點 labels.namespace；節點 ID 為 <cluster>/<namespace>/<service>
service 是 索引 key；Service 節點名稱；與 DNS 兩段式 host 的 service 段比對
cluster_ip 是 ClusterIP Service 的 ipaddress；值 "None" 表示 Headless，省略 IP

kube_endpointslice_labels

Label是否必要圖譜建構用途
cluster 是 slice→service join 的 key 之一
namespace 是 slice→service join 的 key 之一
endpointslice 是 EndpointSlice 資源名稱；與 kube_endpointslice_endpoints join
label_kubernetes_io_service_name 是（需 allowlist） 此 slice 所屬 Service 名稱；缺少則無法建立 EndpointsByService

kube_endpointslice_endpoints

Label是否必要圖譜建構用途
cluster 是 endpoint 所屬叢集
namespace 是 EndpointSlice 所在 namespace；join slice→service 時使用
endpointslice 是 對應到 kube_endpointslice_labels，取得 owning Service
address 合約要求 KSM 標準欄位；目前圖譜建構邏輯未直接使用 IP，但 exporter 必須提供
hostname 是 Headless 三段式 DNS 解析：比對 <pod-hostname> 以找到正確後端 Pod
targetref_kind 是 只接受 Pod（或非空且非 Pod 則略過該 endpoint）
targetref_name 是 後端 Pod 名稱；與 topology 的 PodsByNameNS join
targetref_namespace 建議 Pod 所在 namespace；若空則 fallback 為 EndpointSlice 的 namespace
共通 external label：所有 KSM 系列（含上述三個）都應帶 cluster external label， 以支援多叢集 VictoriaMetrics 匯總。若缺失，reader 會 bucket 成 "unknown"。

Topology 索引如何建立

在 ReadTopology 中，三個指標系列與既有 pod 指標一起解析，產生以下索引（僅記憶體結構，非圖節點）：

kube_service_info └─► ServicesByNameNS[(cluster, namespace, service)] → { cluster_ip } kube_endpointslice_labels └─► sliceToService[(cluster, namespace, endpointslice)] → service_name │ ▼ join kube_endpointslice_endpoints + PodsByNameNS（來自 kube_pod_info） └─► EndpointsByService[(cluster, namespace, service)] → [{ Pod, hostname }, ...] kube_pod_info（既有） └─► PodsByNameNS[(cluster, namespace, pod-name)] → PodNode （Headless fallback：pod 名稱 == DNS hostname 時使用）

Join 邏輯摘要

  1. kube_endpointslice_labels：以 (cluster, namespace, endpointslice) 對應 Service 名
  2. kube_endpointslice_endpoints：同一 key 找到 slice，再取 targetref_* 指向 Pod
  3. Pod 必須已存在於 PodsByNameNS（由 kube_pod_info 建立）；否則該 endpoint 被略過
  4. 最終以 (cluster, namespace, service) 聚合為 EndpointsByService

連線字串解析（D29）如何使用這些索引

觸發條件：traces_service_graph_request_total 某側的 client_k8s_pod_uid 或 server_k8s_pod_uid 為空， 且對應的 client / server label 含 ://。

解析流程

  1. 將 label 當 URL 解析，取出 host
  2. 剝除可選的 .svc.<cluster-domain> 後綴（如 .svc.cluster.local）
  3. 依 dotted label 數量分類：
Host 形狀語意使用的索引 / 指標圖譜結果
<service>.<namespace>（2 段） ClusterIP / Service DNS ServicesByNameNS + EndpointsByService type="service" 節點 + 按需 service-selects-pod 邊
<pod>.<service>.<namespace>（3 段） Headless 逐 Pod DNS 先比對 EndpointsByService[].hostname，再 fallback PodsByNameNS 直接連到真實 Pod 節點（不建立 service 節點）
其他 / topology 找不到 外部 URL 或未知名稱 — type="others"，id="others/<原樣 label>"
為什麼需要 hostname？ KSM 不匯出 Pod 的 spec.hostname。Headless StatefulSet 可能使用自訂 hostname， 此時 DNS 第一段與 Pod 名稱不同；kube_endpointslice_endpoints.hostname 是唯一能正確對上的欄位。 若 hostname 比對失敗，才 fallback 到「pod 名稱 == DNS 第一段」的慣例。
為什麼需要 cluster_ip？ Service 節點的 ipaddress 來自 kube_service_info.cluster_ip。 Headless（cluster_ip="None"）沒有 Cluster IP，故刻意省略 ipaddress， 與 Pod / Node 的 IP 呈現規則一致（IP 放在 typed 欄位，不在 labels 裡）。

圖譜輸出長什麼樣

Service 節點（按需物化）

  • type："service"
  • id：<cluster>/<namespace>/<service>
  • labels：{ "cluster": "...", "namespace": "..." }（嚴格 map[string]string）
  • ipaddress：來自 cluster_ip；Headless 時省略
  • Cytoscape 複合節點：parent = cluster/<cluster>（cluster 群組下）

service-selects-pod 邊（按需物化）

  • 方向：Service → Pod（有向）
  • 來源：EndpointsByService 中該 Service 的每個已解析後端 Pod
  • 永遠叢集內（intra-cluster）
  • 邊 labels 可含 namespace
  • 只在連線字串解析命中 Service 層級時建立；不會為未引用的 Service 批量輸出

PromQL 查詢

在請求時間窗 [start, end] 內，topology reader 對三個系列執行：

last_over_time(kube_service_info[<window>])
last_over_time(kube_endpointslice_endpoints[<window>])
last_over_time(kube_endpointslice_labels[<window>])

若設定 KSG_METRIC_PREFIX（或 --metric-prefix）， 上述 metric 名稱前會加上前綴（例如 o11y_kube_service_info）。 不套用於 traces_service_graph_* 或 up{}。

這三個查詢與其他 5 個 topology 查詢（pod、node、PVC 等）在 errgroup 中平行執行， 共 8 個 PromQL fan-out。

KSM 必要設定

1. 啟用 resources

KSM 必須匯出 services 與 endpointslices 資源（透過 --resources 或等效 allowlist）。

ServiceAccount 需具備對 services、endpointslices 的 list/watch RBAC。

2. EndpointSlice label allowlist（關鍵）

--metric-labels-allowlist=endpointslices=[kubernetes.io/service-name]

沒有此設定，KSM 不會在 kube_endpointslice_labels 上輸出 label_kubernetes_io_service_name， 導致 slice 無法對應 Service —— 即使 kube_service_info 存在， service-selects-pod 邊也會缺失，連線字串只能退回 others。

3. 多叢集 external label

Scrape 設定應為每個叢集的 KSM 加上一致的 cluster external label，供索引與跨叢集拓撲使用。

指標缺失時的行為（優雅降級）

Service / EndpointSlice 查詢是 best-effort 的：

  • 上游未匯出這三個系列 → 索引為空，不會讓整次 graph build 失敗
  • 缺少 label_kubernetes_io_service_name → slice 無法 join Service，該 Service 的 endpoint 被略過
  • endpoint 的 Pod 不在 kube_pod_info → 該 endpoint 被略過
  • 連線字串無法解析 → others/<label> 節點（與 D29 之前行為相容的可見性）

因此 D29 功能對 topology 表面是加法：有指標就解析 Service，沒有就退回 others。

相關指標（非 KSM，但與 Service 解析搭配）

Service/Endpoint 指標只解決「連線字串 → 叢集內 Service/Pod」；完整呼叫圖還需要：

指標來源關鍵 label用途
traces_service_graph_request_total Alloy / Tempo servicegraph client, server, client_k8s_pod_uid, server_k8s_pod_uid, cluster 產生 pod-calls-pod 邊；當 UID 為空且 label 含 :// 時 觸發 D29，才會用到 Service/Endpoint 索引
kube_pod_info KSM cluster, namespace, pod, uid, … 建立 PodsByNameNS / PodsByUID； EndpointSlice 的 targetref_* 必須 join 到這裡的 Pod

Service-graph 查詢層過濾（D30）

client!~"user|unknown",server!~"user|unknown" 會在 PromQL 層排除 servicegraph 的虛擬 peer。這與 Service/Endpoint 指標無關，但影響哪些邊會進入解析管線。
