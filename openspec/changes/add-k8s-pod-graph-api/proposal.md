## Why

Operators running fleets of Kubernetes clusters need a single JSON view of pod-to-pod, pod-to-node, and **cross-cluster** pod-to-pod relationships. Existing tools either stop at the service abstraction, are scoped to one cluster, or require ad-hoc PromQL against `kube-state-metrics` and a service graph metrics endpoint, which is too slow for an interactive UI and does not naturally render cross-cluster edges.

Modern service meshes and OTLP collectors already emit pod-UID-resolved trace metrics that carry the call's pod UIDs (`client_k8s_pod_uid`, `server_k8s_pod_uid`, …) plus a single trace-source `cluster` label (the cluster running Tempo's metrics-generator). When all clusters' metrics flow into a centralised VictoriaMetrics, this is enough information to render a unified, cross-cluster pod-level service graph — provided someone joins each metric series with topology by pod UID (recovering server-side cluster from the topology pod-UID index) and serves it efficiently.

## What Changes

- Introduce a Go (Gin) REST API server, `kube-state-graph`, that returns a unified nodes-and-edges JSON document covering pod, node, and PVC topology for **one or more Kubernetes clusters** and the pod-UID-resolved RPC edges that connect them — including edges that **cross cluster boundaries**.
- Read all inputs from a single centralised VictoriaMetrics endpoint via the Prometheus HTTP API. Every upstream series — both `kube-state-metrics` topology series and `traces_service_graph_*` series — is expected to carry a single `cluster` external label identifying its source cluster (for service-graph metrics, the cluster running the trace producer; the remote/server-side cluster is recovered at build time from the topology pod-UID index).
- Build graphs **on demand** for caller-specified `[start, end]` time ranges, using PromQL `@` timestamp modifiers and range-aware functions (`last_over_time`, `rate`).
- Build the graph **on every request** (no in-process result cache, no singleflight). Each request runs a fresh PromQL fan-out against centralised VictoriaMetrics. Filtering, cluster-selection, and traversal are applied as a projection over the freshly built graph. A future iteration is expected to introduce a horizontally scalable cache mechanism for distributed deployment; that mechanism is **out of scope** for this proposal.
- Resolve service-graph endpoints whose label is a **connection string** (detected by a built-in `"://"` substring check, evaluated per endpoint only when that endpoint's pod UID is empty). The host is parsed as a Kubernetes `.svc` DNS name and resolved against topology: a headless-service string (`<pod-hostname>.<service>.<namespace>`) resolves to the **real backing pod** node (the edge then carries the client cluster); a ClusterIP / service-level string (`<service>.<namespace>`) resolves to a new `type="service"` node plus `service-selects-pod` edges materialised on demand to each backing pod. Both are sourced from `kube_service_info` and `kube_endpointslice_endpoints` topology indexes. When the host is not a parseable in-cluster `.svc` name, or the service / pod is absent from topology, or the cross-cluster match is ambiguous, the endpoint falls back to an `others/<label>` node (`type="others"`, verbatim `name`, `labels={}`) so genuinely external URLs and unknown names stay visible.
- Remove the `KSG_OTHERS_NAME_PATTERN` knob (env var, `--others-name-pattern` flag, and config field) entirely in favour of the built-in `"://"` connection-string detection above. `type="others"` now means a recognised `"://"` connection string that did not resolve to an in-cluster pod or service (a declared external dependency); `type="external"` remains the missing-UID endpoint whose human label is **not** a URL (the producer-regression signal). The two ID namespaces (`others/<label>` vs `external/<label>`) stay disjoint.
- Surface every node's observed IP addresses on a typed top-level `ipaddress` attribute (`string[]`, `omitempty`). `type="pod"` nodes carry `kube_pod_info.pod_ip`; `type="node"` nodes carry the K8s node's `ExternalIP` from `kube_node_status_addresses`; `type="service"` nodes carry `kube_service_info.cluster_ip` (omitted for headless services where `cluster_ip="None"`). `type="pvc"`, `type="external"`, and `type="others"` omit the attribute. `labels.pod_ip` / `labels.host_ip` / `labels.external_ip` are intentionally not emitted — `labels` remains strictly typological metadata.
- Expose the graph in **Cytoscape.js** JSON shape on `GET /v1/graph`.
- Provide cluster discovery (`GET /v1/clusters`) and a static edge-type catalogue (`GET /v1/edge-types`) so callers can populate filter dropdowns without reading documentation.
- Authenticate `/v1/*` and `/debug/*` requests with an **`X-API-Key`** header. Accepted keys come from `--api-keys-file` (one per line, supports K8s `Secret` mounts with hot reload) or `--api-keys` (comma-separated, dev-only). Health probes (`/livez`, `/readyz`), `/metrics`, and the OpenAPI / Scalar UI routes (`/openapi.*`, `/docs`, `/docs/assets/*`) are exempt. Auth is **disabled** when neither flag is set, preserving the dev-friendly default.
- Use Go's standard library `log/slog` for structured logging across the server, with an **OpenTelemetry slog handler bridge** (`go.opentelemetry.io/contrib/bridges/otelslog`) that mirrors every log record to the OTLP logs pipeline. Logs and traces are correlated via `trace_id` / `span_id` fields injected from the active span context.
- Emit **OpenTelemetry traces** for every HTTP request (`otelgin` middleware), every `Builder.Build` invocation, every PromQL query (one span per `errgroup` fan-out leg), and every projection / serialisation pass. Spans carry semantic-convention attributes (`http.*`, `db.system=prometheus`, `db.statement` for the PromQL string, `kube.cluster.count`, `graph.node.count`, `graph.edge.count`) and propagate W3C `traceparent` headers from inbound callers.
- Export traces and logs via **OTLP** (gRPC by default, HTTP/protobuf optional) using the standard OTel Go SDK and `otlptrace` / `otlplogs` exporters. Endpoint, protocol, headers, and sampling are configured exclusively via the OTel-standard environment variables (`OTEL_EXPORTER_OTLP_*`, `OTEL_SERVICE_NAME`, `OTEL_RESOURCE_ATTRIBUTES`, `OTEL_TRACES_SAMPLER`, `OTEL_TRACES_SAMPLER_ARG`) — no bespoke `--otlp-*` flags. Telemetry is **disabled by default**: when `OTEL_EXPORTER_OTLP_ENDPOINT` is unset the SDK installs a no-op tracer/logger and incurs zero export overhead. A graceful-shutdown flush bounded by the existing shutdown grace period drains in-flight spans / logs on `SIGTERM`.
- Run **CI integration tests via testcontainers-go**: a real VictoriaMetrics container started from inside `go test`, with synthetic series injected directly via VM's `/api/v1/import/prometheus` endpoint, and the API server run in-process against the container's URL. No Kubernetes, no scrape pipeline — fast, deterministic, runs on every PR. These integration tests are the **sole** verification path for multi-cluster, cross-cluster, and service-graph code paths.
- Enforce a curated **static-analysis suite** in CI covering complexity, security, error handling, performance, and modern Go idioms, plus **`govulncheck`** for dependency-vulnerability scanning on every PR.
- Auto-generate an **OpenAPI 3.0 specification** from annotated handlers via `swaggo/swag` v2, served at `/openapi.yaml` and `/openapi.json`. Serve the **Scalar API Reference** UI at `/docs` with all assets **vendored** in the binary so the documentation works in air-gapped or otherwise isolated networks (no CDN dependency at view time). A drift gate in CI fails any PR where the generated spec is out of sync with the annotations.
- Adopt **`stretchr/testify`** as the single test-assertion library across the whole repository, with `testify/suite` driving the testcontainers-based integration tests' shared container lifecycle and `testifylint` policing usage in the static-analysis suite.

## Capabilities

### New Capabilities

- `graph-api`: HTTP API surface (Gin) that returns the combined cross-cluster pod / node / PVC / service graph as Cytoscape.js JSON, with time-range parameters, filtering, partial-graph traversal, edge-type discovery, and cluster discovery. The node-type set includes a `type="service"` node (a ClusterIP / service-level connection-string target) and the edge-type catalogue includes a `service-selects-pod` edge (directed service→pod, intra-cluster).
- `cluster-topology-source`: Reader that issues PromQL queries against centralised VictoriaMetrics for `kube_pod_info`, `kube_node_info`, `kube_node_status_addresses`, `kube_pod_spec_volumes_persistentvolumeclaims_info`, `kube_node_labels`, `kube_service_info`, `kube_endpointslice_endpoints`, `kube_endpointslice_labels`, etc., honouring the per-source `cluster` external label and assembling per-cluster pod / node / PVC entities keyed by `(cluster, pod-uid)` and `(cluster, node-name)`. It additionally builds service / endpoint indexes (`ServicesByNameNS`, `EndpointsByService` joining endpointslices to topology pods, `PodsByNameNS`) used to resolve connection-string service-graph endpoints; service nodes and `service-selects-pod` edges are materialised on demand by the service-graph reader rather than emitted wholesale.
- `pod-service-graph`: Pod-UID-scoped service graph reader: PromQL queries against centralised VictoriaMetrics for `traces_service_graph_*` series labelled with `cluster` (trace source / client-side cluster), `client_k8s_pod_uid`, and `server_k8s_pod_uid`. The reader joins the client side via `(cluster, client_k8s_pod_uid)` against topology and the server side via the global topology pod-UID index lookup of `server_k8s_pod_uid`, producing typed edges between pod and node graph nodes — including cross-cluster edges where the resolved source-pod cluster differs from the resolved target-pod cluster. When an endpoint's pod UID is empty and its label is a `"://"` connection string, the reader runs connection-string resolution against the topology service / endpoint indexes: a headless-service string resolves to the real backing pod, a ClusterIP / service-level string resolves to a `type="service"` node with on-demand `service-selects-pod` edges to its backing pods, and an unresolvable string falls back to an `others/<label>` node.
- `container-integration`: CI-driven integration tests using testcontainers-go to start a single VictoriaMetrics container per test package, inject synthetic series via VM's HTTP import API, run the API server in-process, and assert the wire shape and behaviour of every `/v1/*` endpoint against a real PromQL backend. These tests are the sole verification path for multi-cluster, cross-cluster, and service-graph code paths.
- `static-analysis-suite`: Repository-level static-analysis policy: a curated `golangci-lint` configuration covering complexity, security, error-handling, performance, modern-Go-idiom checks, and `testifylint`; a `swag init` drift gate; plus `govulncheck` runs on every PR to detect dependency vulnerabilities. All run in CI and gate merges.
- `api-docs`: Auto-generated OpenAPI 3.0 specification (via `swaggo/swag` v2) served at `/openapi.yaml` and `/openapi.json`, plus a Scalar API Reference UI served at `/docs` with all JS / CSS assets vendored into the Go binary so documentation works offline / air-gapped. A drift contract test asserts every Gin route is documented and every documented path exists.
- `otlp-observability`: OpenTelemetry tracing and logging pipeline. `otelgin` middleware spans every `/v1/*` request; manual spans wrap `Builder.Build`, each parallel PromQL query in the topology / service-graph errgroups, projection, and serialisation. `log/slog` is bridged to OTLP logs via `otelslog`, with `trace_id` / `span_id` correlation. Configuration is OTel-standard environment variables only (`OTEL_EXPORTER_OTLP_*`, `OTEL_SERVICE_NAME`, `OTEL_RESOURCE_ATTRIBUTES`, `OTEL_TRACES_SAMPLER`); telemetry is no-op when `OTEL_EXPORTER_OTLP_ENDPOINT` is unset. Graceful shutdown flushes pending exports within the existing shutdown grace period.

## Impact

- New Go module with Gin, Prometheus Go client, `golang.org/x/sync/{errgroup,semaphore}`, `log/slog`, and the OpenTelemetry Go SDK (`go.opentelemetry.io/otel`, `otlptrace`/`otlplogs` exporters, `otelgin`, `otelslog`) as primary dependencies. VictoriaMetrics is consumed over HTTP via the Prometheus query API and is not vendored. No `client-go`, no informers, no Kubernetes API access.
- New HTTP API surface (`/v1/graph`, `/v1/clusters`, `/v1/edge-types`, `/v1/livez`, `/v1/readyz`, `/metrics`, optional `/debug/last-queries`) that downstream UIs and scripts will depend on.
- New verification artefacts under the repo:
  - `internal/integration/` — CI-driven testcontainers tests.
- New static-analysis artefacts (`.golangci.yml` enabling the curated linter set, CI workflow steps for `golangci-lint` and `govulncheck`).
- Each upstream cluster operator becomes responsible for ensuring its scrape pipeline applies the `cluster` external label uniformly across `kube-state-metrics` and service-graph metrics. This is documented but not enforced in code.
- No existing code paths or specs are modified.
